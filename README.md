## Releases

Pushing a SemVer tag like `v0.1.0` runs `.github/workflows/release.yml`, stamps `package.json`, `src-tauri/tauri.conf.json`, and `src-tauri/Cargo.toml` with the tag version for that CI build, builds the Tauri app for Windows, macOS, and Linux, publishes a GitHub Release, and uploads the updater `latest.json` generated by `tauri-apps/tauri-action`.

The workflow expects the platform signing secrets shown in GitHub, plus these updater signing secrets:

- `TAURI_SIGNING_PUBLIC_KEY`
- `TAURI_SIGNING_PRIVATE_KEY`
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` (optional, but keep it set if the key was generated with a password)

Generate the updater key pair with:

```sh
bun tauri signer generate -w ~/.tauri/all-i-care-about.key
```

Store the public key output in `TAURI_SIGNING_PUBLIC_KEY`, and store the private key content or path in `TAURI_SIGNING_PRIVATE_KEY`.

## Provider Security Model

Provider settings are intentionally split between UI-owned metadata and Rust-owned trust decisions.

- The frontend renders provider setup fields and computes candidate provider origins from plugin metadata.
- Rust owns all settings reads and writes through Tauri commands; the frontend does not have direct Tauri Store access.
- Secret provider fields, such as personal access tokens, are encrypted in Rust with AES-256-GCM.
- The AES key is generated by Rust and stored with Tauri keyring.
- Existing secret values are never returned to the frontend during provider editing.

Provider HTTP requests also run through Rust:

- JavaScript provider implementations call `providerFetch` with a provider id and optional secret injection metadata.
- JavaScript never receives decrypted provider secrets.
- Rust loads the provider, verifies its security seal, validates the request origin, decrypts the secret, injects the header, and performs the request.
- Rust validates the request origin before decrypting or injecting any secret.

Each saved provider has a `security` snapshot:

- `allowedOrigins` is the visible, normalized list of origins the provider may contact.
- `sealed` is an AES-GCM authenticated payload created by Rust.
- The sealed payload contains the provider id, provider type, allowed origins, trusted secret setting paths, and security version.

The visible origins are not secret, but they are tamper-evident:

- Editing `security.allowedOrigins` manually invalidates the seal.
- Editing the sealed blob invalidates the seal.
- `providerFetch` rejects providers whose visible security data does not match the sealed payload.

Dynamic provider origins must come from fields marked with `originAccess: true`.

- Only `url` fields may use `originAccess`.
- `originAccess` URL fields must use HTTPS.
- Static plugin origins and dynamic `originAccess` origins are combined and normalized when the provider is saved.
- If the normalized origin list changes, Rust clears any preserved old secrets unless the user entered replacement secrets in the same save.

Enabled provider capabilities are not part of the security seal. Changing capabilities such as `GetPR` or `GetIssue` does not clear provider secrets or require a new token.

## Local OpenSearch Logging Sandbox

Use the local Docker Compose stack to test an OpenSearch logging provider with a single-node OpenSearch cluster, OpenSearch Dashboards, and a dummy application that emits debug, info, warning, and error logs every second.

```sh
docker compose -f docker-compose.opensearch.yml up --build
```

- OpenSearch: http://localhost:9200
- OpenSearch Dashboards: http://localhost:5601
- Write/read alias: `dummy-app-logs`
- Backing index: `dummy-app-logs-000001`
- Index template: `dummy-app-logs-template`
- Dashboards data view: `dummy-app-logs`

The bootstrap containers create the `dummy-app-logs` OpenSearch write alias and a matching Dashboards data view with `@timestamp` as the time field before the logger starts.

To reset the dummy cluster data:

```sh
docker compose -f docker-compose.opensearch.yml down -v
```
